# Обучение и хранение

## Формат правки

`record-feedback` принимает JSON-файл:

```json
{
  "locator": { "jid": "чат", "day": "Вчера", "time": "19:08" },
  "platform": "claude | codex | other",
  "initial":    { "тема": "...", "срочность": "низкая", "нужен_ответ": false },
  "correction": { "срочность": "высокая", "нужен_ответ": true },
  "scope": { "type": "chat", "value": "<jid или ключевое слово>" },
  "authored_by_lpr": true,
  "pattern": {
    "title": "чат прораба — всегда высокий приоритет",
    "description": "одна фраза: когда правило применимо",
    "content": "1-3 предложения: что делать, когда сработало",
    "rule_key": "chat-prorab-urgency",
    "expected": { "срочность": "высокая" },
    "forbidden_actions": []
  }
}
```

Локатор нужен только для HMAC-ссылки. База **никогда** не хранит jid, номер, имя чата,
текст сообщения и вложения — ни в одном поле.

`pattern` опускается, если правка касается только этого сообщения. Если он задан, правило
сразу становится `active_provisional` и попадает в следующий `export-context`.

`expected` — то, что правило навязывает результату. Задавай **только те поля, которые
действительно фиксируешь**: остальные останутся от модели.

## Область действия (scope)

| type | значение | когда |
|---|---|---|
| `message` | — | разовая правка, без обобщения |
| `chat` / `contact` / `group` | jid | «этот чат всегда...» (значение хешируется) |
| `keyword` | слово/фраза | «слово „авария“ — всегда высокая» (хранится как есть, нужно для сопоставления) |
| `project` | код проекта | правило для контура работ |
| `global` | — | для всех чатов, применять осторожно |

`keyword` — единственный scope, где значение лежит в открытом виде: иначе его не с чем
сопоставлять. Не используй его для имён и номеров.

## Жизненный цикл

```text
active_provisional -> active_verified -> retired
                   -> rejected
```

- `active_provisional` — после одной явной правки ЛПР, применяется сразу.
- `active_verified` — после трёх подтверждений (`confirm-pattern`).
- Контрпример фиксируется, правило при этом **не удаляется молча**.
- Снятие или отклонение — только явным решением ЛПР (`retire-pattern` с причиной).

Правила, написанные человеком (`authored_by_lpr: true`), машина переписывать не может —
контракт помечает их `origin=human`, и попытка изменить их не от лица человека падает с
ошибкой. Предложить изменение можно, применить — нет.

## Где лежит состояние

Вне плагина и вне vault. По умолчанию `%LOCALAPPDATA%\Tech77\BipTriage`, переопределяется
`BIP_TRIAGE_STATE_DIR`. HMAC-ключ — рядом с базой, права только пользователю.

Это операционная память, а не принятое знание и не вход для Graphify. Перенос чего-либо
отсюда в vault — отдельное решение ЛПР.

## Схема

Общий контракт обучения скиллов v0.2 (`store_schema.py`) — тот же, что у остальных
скиллов, чтобы одна проверка читала базы всех. Файл вендорится, семантика не форкается.
Специфика BiP — только scope-типы выше и функция применения правил к вердикту триажа.

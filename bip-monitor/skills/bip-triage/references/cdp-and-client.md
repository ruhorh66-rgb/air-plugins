# Механика доступа и особенности клиента BiP

Всё ниже проверено на живом клиенте (SRVLM01, 2026-07-21, BiP 3.0.104). Это не гипотезы —
не переисследуй заново, но проверь, если клиент обновился.

## Что такое BiP Desktop

Electron 22 (Chrome 108) поверх `web.bip.com` — десктоп это обёртка над веб-клиентом.
Единственный page-таргет CDP: `https://web.bip.com/`.

**Официальный BiP Developer API (`Turkcell/BiP.AI`: TES/REX/FTS) личные чаты не отдаёт** —
это модель Service/бота. Поэтому механизм только клиентский.

## Почему не Playwright

`connect_over_cdp` падает: `Browser context management is not supported` — Electron 22 не
реализует `Browser.setDownloadBehavior` (проверено на playwright 1.61.0). Скрипты говорят
с CDP напрямую по websocket (`bip_cdp.py`), нужен только `Runtime.evaluate` и `Input.*`.
Playwright остаётся нужен web-фолбэку.

## Селекторы

Опираться можно на `itemtype` и `data-jid` — остальная разметка это CSS-modules с хешами
(`_contact__content__body_fo3np_19`), которые переживают максимум один релиз.

| Что | Селектор |
|---|---|
| строка списка чатов | `[itemtype="ChatListRow"]` + `data-jid` |
| имя чата / время / превью | `[class*='header__name']` / `[class*='header__time']` / `[class*='message__text']` |
| непрочитанные | `[class*='extra__unread']` |
| своё исходящее (в списке) | `[class*='message__icon'] svg` — галочка статуса |
| сообщение в ленте | `[itemtype="TextBubble"]`, также `ImageBubble` / `RichLinkBubble` / `ContactBubble` |
| текст сообщения | `.readonly-editor` внутри пузыря |
| время сообщения | `[class*='message_status__time']` |
| своё исходящее (в ленте) | `[class*='message_status__icon']` |
| разделитель дня | `[itemtype="date-bubble"]` |
| поле ввода | `[data-lexical-editor][contenteditable=true]` |

Если разметка поехала — `bip_discover.py` печатает структуру и длины (без текста) и по ней
подбираются новые селекторы.

## Четыре особенности, на которых легко ошибиться

1. **Список чатов несёт ПОЛНЫЙ текст последнего сообщения.** Обрезка чисто CSS-ная
   (`scrollWidth` 509 при `clientWidth` 292). Поэтому режим списка вообще не открывает чаты.
2. **Порядок в DOM не хронологический.** Подгрузка истории дописывает старые узлы в конец,
   поэтому лента сортируется по фактическому положению на экране.
3. **BiP не подставляет даты.** `date-bubble` содержит плейсхолдеры локали — `ДД.ММ.ВЕК`,
   `ддд`; корректно только «Вчера». Дату сообщения из интерфейса взять нельзя.
4. **Стабильного `message_id` нет** — у пузыря только `itemtype` и классы. Дедуп в режиме
   чата идёт по (jid, день, время, текст), с дырой из-за п. 3: одинаковый текст в одно
   время в разные старые дни схлопнется.

## Локальная база клиента пуста

`%APPDATA%\BiP` (IndexedDB, Local Storage, WebStorage, databases) держит только список
чатов, имена и аватары — **истории переписки на диске нет**, она приходит с сервера при
открытии чата (`GetChatHistory`, XMPP поверх websocket). Читать прошлое «ничего не
открывая» из файлов невозможно — проверено, не трать на это заход.

Непроверенная альтернатива: пассивная подписка на `Network.webSocketFrameReceived` дала бы
все входящие целиком в реальном времени без открытия чатов. Требует решения ЛПР.

## Отправка

Поле ввода — Lexical-редактор: присваивание значения он игнорирует. Текст вводится через
CDP `Input.insertText`, отправка — `Input.dispatchKeyEvent` с Enter.

Обязательный порядок (`bip_send.py` его соблюдает): открыть чат и назвать его → вставить
текст → **прочитать поле обратно и сверить до Enter** → Enter → подтвердить фактом (поле
очистилось, последний пузырь исходящий нужной длины). Не верить тому, что команда прошла —
проверять состояние.

# BiP triage

Мониторинг собственных рабочих чатов BiP локальной моделью, с накоплением правил из
правок ЛПР. Ничего не отправляет, пока ЛПР не скажет отправить конкретный текст.

## Перед первым запуском

1. BiP Desktop установлен и залогинен, порт отладки поднят:
   ```powershell
   Get-Process BiP* | Stop-Process -Force       # трей тоже
   & 'C:\Program Files\BiP\BiP.exe' --remote-debugging-port=9222 --remote-allow-origins=*
   irm http://localhost:9222/json/version       # должен вернуть Browser/webSocketDebuggerUrl
   ```
2. Локальная LLM слушает `http://localhost:8080/v1` (llama.cpp/Qwen или совместимое).
3. `pip install websocket-client`; store обучения: `python learning_store.py init`.

Клиент запускает ЛПР — не запускай BiP сам, если он не запущен: спроси.

## Что делать по запросу

### Триаж непрочитанного (по умолчанию)

```powershell
python scripts/bip_desktop_monitor.py --once          # один проход
python scripts/bip_desktop_monitor.py --minutes 15    # следить
```

Читает **список чатов**, ничего не открывая. Строго read-only: собеседники ничего не
видят, счётчики непрочитанных не трогаются. Видно последнее сообщение каждого чата.

Результат — в `E:\-4-\bip\triage.jsonl`. В консоль идут только хеш чата, длины и вердикт:
содержимое и номера наружу не выводятся.

### Разбор конкретного чата

```powershell
python scripts/bip_desktop_monitor.py --chat "фамилия" --limit 50
```

> Открытие чата **помечает сообщения прочитанными** — собеседник видит отметку, счётчик
> обнуляется. Это не read-only. Запускать только когда ЛПР явно просит разобрать чат;
> если он просто спросил «что нового» — хватит режима списка. Штатное место режима —
> аккаунт ИИ-ассистента, где отметки ожидаемы.

### Отправка сообщения

```powershell
python scripts/bip_send.py --chat "фамилия" --text "точный текст" --yes
```

Только по явной команде ЛПР и только с текстом, который он подтвердил дословно. Если
текст не задан — предложи формулировку и дождись подтверждения, не выдумывай за него.
Автоответы, реакции на входящие и рассылки — **не разрешены**: это отдельное решение ЛПР,
которого по умолчанию нет.

Разрешение действует на **одну** отправку: «отправь X» не открывает право дослать
исправленную версию — доложи и жди команды.

### Отправка файла

```powershell
python scripts/bip_send.py --chat "фамилия" --file "E:\путь\документ.pdf" --yes
```

Вложение уходит **не по Enter**, а кликом по стрелке `a.send-media-button` в блоке
предпросмотра `div.send-media`. Прикрепление проверяется существованием этого блока:
`input.files.length` остаётся нулём, потому что BiP переносит файл в своё состояние и
чистит input — судить по нему нельзя (`ERR-2026-000068`).

### Удаление своего сообщения

```powershell
python scripts/bip_send.py --chat "фамилия" --delete "часть текста или имя файла" --yes
```

Необратимо, у собеседника тоже. Только по явной команде и только на своё сообщение.
Механика: hover по пузырю → «три точки» в правом верхнем углу (клик по координате, не по
селектору) → «Удалить» → «Удалить от всех» либо «Удалить у меня». Кнопка «Сдавайся» в
подтверждении — это машинный перевод «Отмена», не нажимать. Факт проверяется исчезновением
пузыря: диалог может остаться на экране после успешного удаления.

## Обучение на правках

Когда ЛПР поправляет вердикт («это не срочно», «этот чат всегда высокий приоритет»,
«слово „авария“ — всегда высокая»), правка сохраняется правилом и применяется дальше.

```powershell
python scripts/learning_store.py record-feedback --input feedback.json
python scripts/learning_store.py export-context          # действующие правила
python scripts/learning_store.py confirm-pattern --pattern-id BIP-PAT-000001
python scripts/learning_store.py retire-pattern --pattern-id BIP-PAT-000001 --reason "..."
```

Формат `feedback.json`, область действия правил и жизненный цикл — в
[references/learning-and-storage.md](references/learning-and-storage.md).

Правила применяются **после** модели и переопределяют только те поля, которые задают:
правка ЛПР должна побеждать модель детерминированно, а не «влиять на промпт».

Два правила гигиены обучения:

- Не заводи правило из одной догадки. Правило создаётся из **явной** правки ЛПР, а не из
  того, что модель сочла закономерностью.
- Правило не удаляется — снимается с причиной (`retire-pattern`). Контрпример
  фиксируется, а не затирает историю.

## Границы

- **Только собственный аккаунт ЛПР** (или заведённого им ассистента). ToS-риск принят.
- **ПДн третьих лиц**: сырьё и триаж — только вне git и vault (`E:\-4-\bip\`), store
  обучения — в `%LOCALAPPDATA%\Tech77\BipTriage`, и он не хранит ни текстов, ни номеров,
  ни jid (только HMAC). В vault/RAG попадает лишь санитизированный итог по отдельному
  решению.
- **Не выводи содержимое чатов в ответ модели**, если ЛПР не потребовал этого явно: это
  отправка переписки в облако. По умолчанию отдавай счётчики и вердикты; когда нужно
  показать текст — отдавай файлом с машины.
- **Никаких write за пределами отправки по команде**: не отвечай за ЛПР, не помечай
  прочитанным ради удобства, не удаляй и не архивируй чаты.

Подробности механики и ограничений клиента —
[references/cdp-and-client.md](references/cdp-and-client.md).

## Если что-то не работает

- `localhost:9222` молчит → BiP запущен без флага или висит в трее; перезапустить по шагам выше.
- Playwright не цепляется → так и должно быть, у Electron 22 нет `Browser.setDownloadBehavior`;
  скрипты ходят в CDP напрямую, Playwright нужен только web-фолбэку.
- Пусто в списке чатов → разметка клиента поехала после обновления; снять заново
  `python scripts/bip_discover.py` и поправить селекторы в шапке монитора.
- Не выдумывай обход, если упёрся — зафиксируй блокер и спроси ЛПР.
